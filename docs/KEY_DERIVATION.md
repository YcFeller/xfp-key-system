# 解锁密码（密钥）派生 — 设计说明与开源骨架

## 1. 功能在系统中的位置

「解锁密码」用于用户在手表端输入、以解锁已购表盘资源或快应用能力。本仓库中，派生结果会在以下流程中出现：

- 激活 API（[`app/api.php`](../app/api.php)）：按订单关联的 `plan_id` 查询 `xfp_wflist`（表盘）或 `xfp_quickapp_list`（快应用），对每个产品 ID 生成一条解锁密码。
- 用户中心订单激活（[`app/user/user_orders_api.php`](../app/user/user_orders_api.php)）：按订单下所有表盘 ID 生成密码（当前实现仅遍历表盘表）。
- 快捷工具（[`app/shortcut_tool_api.php`](../app/shortcut_tool_api.php)）：单笔设备码 + 表盘 ID 试算等场景。

业务上还会把「订单号、产品 ID、设备码、解锁密码」等写入 `xfp_activation_records` 用于审计与防重复，**派生算法须与你在设备端或表盘包内约定一致**，否则用户无法解锁。

## 2. 历史参考设计（本仓库默认代码中**不再包含**实现）

以下为早期线上曾采用的一种**示例级**思路，仅作文档说明，便于你理解字段含义与改造方向。**开源仓库中的 PHP 已改为骨架，不会按下列步骤自动算出可上线密钥。**

### 2.1 输入

| 字段 | 含义 |
|------|------|
| 设备码 | 用户从手表等设备复制的 PSN / 设备标识字符串 |
| 产品 ID | `watchface_id` 或 `quickapp_id`（字符串，来自业务表） |

### 2.2 拼接常量路径模板（历史行为）

将设备码与产品 ID 嵌入一段**固定格式的路径字符串**（与表盘市场路径相关），例如：

```text
{device_code}/data/app/watchface/market/{product_id}//_lua/app/
```

> 说明：旧代码中，快应用激活分支虽走 `xfp_quickapp_list`，但派生时仍复用了与表盘相同的路径模板（历史遗留；你若区分表盘/快应用，应在自实现中分别约定不同模板或算法）。

### 2.3 哈希与截断

- 对上述整段字符串做 **SHA-256**，得到十六进制摘要；
- 取摘要十六进制字符串的**末尾 6 个字符**作为中间串。

### 2.4 `convert_to_odd_numbers`（历史辅助步骤）

对「末尾 6 字符」逐字符处理：

- 若为 **小写字母 a–z**：先映射为 `(ord(c) - ord('a') + 1) % 10`，若为偶数则加 1 变为奇数，再作为一位数字字符输出；
- 其余字符（如数字、或其它十六进制字符）**原样保留**。

最终字符串即历史上作为「解锁密码」返回给前端的值。
旧版加密模式参考了项目“Watchface Locker”

## 3. 开源仓库中的实际行为（骨架）

- 所有派生统一入口为 [`app/UnlockKeyDerivation.php`](../app/UnlockKeyDerivation.php) 中的 `UnlockKeyDerivation::derive()`。
- **默认实现会抛出异常**，不包含上一节的任何具体算法，避免公开环境与你的设备端协议强行绑定。
- **你必须**在该文件中自行实现与你的产品、设备、表盘包一致的规则（可仍为哈希+裁剪，或对称密钥、HMAC、与云端协商的 token 等）。

实现完成后，请自行完成联调（同一组 `device_code` + `product_id` 在服务端与手表端得到相同解锁结果）。
